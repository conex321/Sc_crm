**1. `lib/crm/opportunities.ts` — list loader:**
```ts
export type ListSort = DealSort | "title" | "stage" | "created" | "updated" | "account";
export async function listDealsForListView(pipelineId: string, filters: DealFilters,
  sort: ListSort, dir: "asc" | "desc", page: number):
  Promise<{ rows: BoardOpportunity[]; total: number; capped: boolean }>
```
Implementation decision (documented — do not "optimize" into `.in()` traps): fetch up to 500 matching deals in one `.range(0, 499)` query (same bound as the board's open cap), merge next-task rows from `opportunity_next_task` filtered by `pipeline_id` (one query — never per-row), sort in server JS (reuse `compareDeals` for the shared keys; add comparators for title/stage-position/created/updated/account-name; apply `dir` by inverting), then slice `[(page-1)*50, page*50)`. `total` = matched count (use `{ count: "exact" }` head request or the fetched length; `capped = total >= 500`). This keeps pagination correct for EVERY sort key — including next-activity and cross-table columns that PostgREST cannot order by — while rendering only 50 rows. If `capped`, the footer appends `(first 500 shown)`.
